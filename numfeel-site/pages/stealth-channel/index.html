Network tab 抓不到的数据通道

浏览器在往外发数据，你却盯着 F12 一个都抓不到。6 种数据外发通道逐个对比，看哪种能被 Network tab 抓到，哪种彻底隐身。

现在打开 F12 → Network → 保持开启

6 条通道，逐个发一条秘密消息

每张卡片会发出一条带唯一 token 的消息。点「发送」后，去 Network tab 找找这条 token——能找到的说明该通道可见，找不到的说明它隐身了。

WebRTC 配对：两个标签页 / 两台设备

WebRTC 要两个端才能建连。新建房间，把链接发给另一个标签页或手机，配对成功后互发消息——然后去 Network tab 翻，你会发现消息根本不在里面。

抓包挑战

点击「开始挑战」，页面会通过 6 个通道各发一条带 token 的秘密消息。去 Network tab 搜 token，看你能抓到几条——有两条注定抓不到。

为什么 WebRTC 抓不到？

01

HTTP 栈被强制记录

DevTools 钩在网络层，所有 fetch / XHR / sendBeacon / 资源加载都逃不掉。这是调试工具的职责，没有 JS API 能关掉它。

02

WebSocket 只记连接

建立连接是 HTTP 升级握手，主列表会有一条 ws 记录。但之后的双向消息帧藏在 Messages 子标签里，不点开连接根本看不到内容。

03

WebRTC 走另一条路

DataChannel 用 SCTP over DTLS，是浏览器到浏览器的 P2P 直连，完全不经过 HTTP 栈。Network tab 根本不钩这条管道，所以数据传输在这里彻底消失。

04

信令可见，数据不可见

建立 P2P 连接前要交换 SDP 和 ICE，这一步通常走 WebSocket（可见）。但连上一旦建立，之后的所有数据都走 P2P 通道——Network tab 只能看到「搭桥」，看不到「运货」。

核心结论

「盯着 Network tab 就能抓到所有数据外发」是个错觉。HTTP 类请求确实逃不掉，但 WebRTC DataChannel 建连后完全在另一条管道上跑。追踪、数据外泄完全可以借这条隐身通道完成——光靠 DevTools 抓不到。

防御者怎么抓隐身通道？

既然 Network tab 抓不到，安全人员还能怎么发现 WebRTC 外发？

Network tab 看不到的

  • DataChannel 的消息内容
  • P2P 传输的实际数据
  • 连接建立后的持续通信
  • 媒体流（音视频）的真实流量

仍能检测的手段

  • RTCPeerConnection.getStats() 查连接状态
  • 劫持 RTCPeerConnection 构造函数，监听页面是否建连
  • 浏览器扩展 / 企业代理在传输层看 SCTP/DTLS 流量
  • CSP / feature-policy 禁用 WebRTC 从源头阻断
