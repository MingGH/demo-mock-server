关键差距在这里：当查询条件超越纯文本搜索（按类型、按时间范围过滤），文件方案必须逐行解析 JSON 并检查字段，复杂度直线上升。而 SQLite 用索引直接定位，几乎不受数据量影响。

场景四：删除消息

撤回/删除消息 — 文件方案需要重写整个文件（去掉被删的行），SQLite 原地 DELETE。

这是文件方案的死穴：删除一条消息 = 读取整个文件 + 过滤 + 重写。文件越大越慢。消息撤回、过期清理在 IM 场景中是高频操作。

云风说的没有「错」，但前提条件很苛刻：只有 append + 全文搜索两个操作，没有删除、没有按时间/类型过滤、没有多设备同步。一旦需求超出这个极简模型，文件方案的代码复杂度和性能都会急剧恶化。微信作为一个功能齐全的 IM 客户端，选择 SQLite 是工程上合理的决策。
