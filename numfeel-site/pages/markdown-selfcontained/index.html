 体积只是表象，这张表才是关键

    
把常见格式摆在一起看。✓ 支持 △ 部分/有条件 ✗ 不支持。没有任何一种格式能同时占满「单文件 + 可 diff + 可文本编辑 + 双击即看」。

    

      

    

    
格式说明参考各自规范：TextBundle、Jupyter nbformat、Pandoc --embed-resources、CommonMark。

  


  
  

    
 那为什么没有一个「赢家」

    

      

        
01

        
纯文本是命根子

        
Markdown 的全部价值在于它是纯文本：能 git diff、能 grep、任何编辑器都能改。塞进 ZIP 或 base64，这些立刻全废。自包含和纯文本天生打架。

      

      

        
02

        
没有「钦定」格式的人

        
Markdown 没有统一标准机构，CommonMark、GFM 各管一摊。DOCX 背后是微软 + ISO，JAR 背后是 Sun/Oracle，单一推手才能形成网络效应。Markdown 没有。

      

      

        
03

        
问题已经被「分工」解决

        
创作态用纯文本 .md + assets（保留 diff 和可编辑），分发态渲染成自包含 HTML 或 PDF。你要的「单文件」其实就是 pandoc --embed-resources 出来的那个 HTML，它一直都在。

      

    

    

      
 一句话

      
base64 内嵌让文件膨胀约 33%（3 字节编码成 4 个字符，数学上跑不掉），还杀死了纯文本属性；ZIP 体积最小却要先解压；自包含 HTML 双击即看却不再是 Markdown。三个方向各有取舍，谁也没法既当源格式又当分发格式——这才是「没有统一标准」的真正原因。
