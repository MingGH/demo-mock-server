跨域请求限制实验室

都说浏览器「严格限制跨域」。到底限制了什么？这一页让你亲手发一个被 CORS 拦下的请求，去 Network 面板看请求其实是 200、响应体就躺在那——只是浏览器不肯把响应交给你的 JS。再切一下服务器的 CORS 策略，看放行是怎么发生的。

实验 2｜预检：OPTIONS 先问一句

带上一个自定义头 X-Demo，这就不再是「简单请求」了。浏览器会先发一个 OPTIONS 问问服务器「我能不能这么请求」，得到允许才发真正的请求。deny 下这个预检会被 403 顶回来，真正的请求根本不会发出；allow 下预检通过，才会发出后续请求。

为什么这件事只能浏览器管

HTTP 无状态。服务器收到一个请求，只能靠 Origin / Referer 两个头去猜这请求是谁指使的。这俩头在浏览器里由浏览器自己写、脚本改不了；可一旦离开浏览器（curl、Python），想填什么就填什么。服务器永远没法可靠判断一个请求是不是某个网页里的脚本偷偷发的。

只有浏览器知道代码的出身

脚本跑在浏览器肚子里，浏览器知道这段代码从哪个源加载来，这个信息脚本自己篡改不了。同源策略只能由浏览器执行，因为它管得了，别人管不了。

恶意脚本用的是你的脸

脚本运行时拥有你的全部浏览器上下文：cookie、内网位置、登录态、IP。打开一个投毒网页，里面的脚本就能以你的身份去戳其它站点。浏览器是这一切的唯一出入口。

只拦 cookie 远远不够

路由器、打印机、内网管理台对「本网段的请求」几乎无条件信任，根本不要 cookie。一个恶意网页就能以你的网络位置去改 192.168.x.x 的 DNS。所以限制得落在「读跨域响应」这一层。

一句话结论

浏览器严格限制跨域，因为它是唯一能可靠识别「代码出身」的角色。限制的落点在「不让脚本读跨域响应」，请求该发还是发、服务器该处理还是处理。这套机制由服务器声明、浏览器执行，本质是 default-deny 加显式 opt-in 的安全范式。CORS 管「读」，SameSite cookie 管「带身份的写」，两条线各管各的，别混成一回事。
