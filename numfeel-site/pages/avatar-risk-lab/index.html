SVG XSS 实战对比

同一份带 <script> 的 SVG 文件，用不同标签加载结果天差地别。这就是为什么 SVG 必须经过服务端解码重绘——只要平台有任何一个角落（导出 PDF / 后台审核页 / 邮件预览）用了非 <img> 加载，就立刻 XSS。

<img src>（安全）
浏览器 sandbox 拦下脚本，SVG 里的 <text> 保持 "SCRIPT_NOT_RUN"。
<object data>（XSS）
点击后看两处变化：① SVG 内文字变成绿色 "✅ 脚本已执行"；② 页面顶部弹出红色横幅（脚本通过 postMessage 通知宿主）。
为什么不是直接弹 alert / 偷宿主 cookie？Chrome 92+ 禁止跨域 iframe/object 弹 alert，跨域读 window.top 也会抛 SecurityError。但同源场景下（比如平台把用户上传的 SVG 内联到自己页面，或者放在同域 CDN 用 <object> 加载），XSS 仍然能完整执行——读 cookie、改 DOM、键盘记录、CSRF 全套。

风险全景

把"用户 URL 当头像"这件事拆开看，至少有这些坑：

钓鱼 401 弹窗

服务端返回 401 + WWW-Authenticate: Basic，浏览器立刻弹原生账号密码框——伪装成「请登录以查看头像」，骗用户输入主站凭据。

重定向到攻击载荷

URL 看着是 .jpg，实际服务端 302 跳到 http://localhost:8080/admin?...，命中内网管理后台、Redis、云元数据等——SSRF 的入门姿势。

内容随时被替换

图片在别人服务器上，今天是正常头像，明天可能换成血腥/色情/政治敏感图片——平台口碑被一张 <img> 拖下水。

访客信息泄露

每个浏览过该头像的用户都把自己的 IP、UA、Referer 上交给攻击者，方便针对性钓鱼或者画像建模。

流量炸弹 / 慢响应

10MB 一张的"头像"或永远不返回的连接，把你的页面、CDN、流量包一起拖下水。

外链失效破图

第三方服务挂了，你的用户列表瞬间一堆碎图，体验崩盘。

正确做法

核心原则只有一条：永远不要直接信任用户提供的 URL。把它当作"原始素材"，做一道完整的清洗工序：

  1. 协议白名单：只允许 https://，拒绝 http / file / data / javascript / gopher。
  2. SSRF 防护：DNS 解析后校验 IP 不在 127.0.0.0/8、10/8、172.16/12、192.168/16、169.254/16（云元数据）等保留段。
  3. 禁用重定向：服务端拉取时关闭 follow redirects，或对每一跳都重新做 IP 校验。
  4. Content-Type + Magic Number 双校验：响应头说 image/jpeg 不够，还要校验文件头 FF D8 FF。
  5. 大小 / 尺寸限制：超过 5MB 或 4096×4096 直接拒绝。
  6. 解码重绘：用 ImageIO / sharp 解码后重新编码输出，剥离 EXIF、隐写 payload、SVG 里的脚本。
  7. 内容审核：接黄反/暴恐/版权检测，留好申诉与回退路径。
  8. 转存自家 CDN：以自有域名对外服务，从此与第三方 URL 一刀两断。
